# 策略0：高隐蔽相对时序 验证说明

## 当前设计

策略0仍然用两个连续包间隔的大小关系表示 1 bit：

- `0`：短间隔 + 长间隔。
- `1`：长间隔 + 短间隔。

本次采用方案B：在每个承载业务包的 UDP 负载开头加入 2 字节同步标签。标签字段包含帧编号、策略编号、相位和符号索引，并经过轻量异或混淆。标签只用于接收端重同步，不承载真实隐蔽数据；真实隐蔽数据仍然由包间隔关系承载。

这样做的目的不是恢复丢掉的那个 bit，而是避免少量丢包后后续全部错位。接收端会把无法判断的位置记为 `unknown_bits`，后续符号仍可继续按索引解码。

## 文件位置

- `results_scheme_b/input_payload.bin`：104bit 测试输入数据。
- `results_scheme_b/input_bits.txt`：输入数据的二进制文本。
- `results_scheme_b/summary.csv`：离线矩阵测试汇总。
- `results_scheme_b/summary.json`：离线矩阵测试详细结果。
- `results_scheme_b/vm_mininet/`：Ubuntu 虚拟机 Mininet/BMv2 实测结果。
- `results_scheme_b/测试报告.md`：本策略测试报告。

## 手动验证流程

1. 启动拓扑：

```bash
cd /home/p4/yws-covert
sudo bash run.sh
```

2. 策略0默认使用 UDP 端口 `50000`。在 Mininet CLI 中启动接收端：

```bash
h2 python3 experiments/live_receive_strategy.py --strategy 0 --iface h2-eth0 --output celue0/results_scheme_b/manual/decoded_output.bin --summary celue0/results_scheme_b/manual/receive_summary.json --timeout 30 --expected-bytes 13
```

3. 在另一个 Mininet CLI 或 xterm 中发送承载业务流：

```bash
h1 python3 experiments/live_send_strategy.py --strategy 0 --input celue0/results_scheme_b/input_payload.bin --dst-ip 10.0.2.2 --src-ip 10.0.1.2 --send-mode socket
```

4. 如需同时验证普通业务流不断，可以开低速 UDP iperf 背景流，注意不要打满链路：

```bash
h1 iperf -u -s -p 51000
h2 iperf -u -c 10.0.1.2 -p 51000 -b 500K -t 60 -i 1
```

## 注意事项

- 不要用满速 TCP 或满速 UDP 背景流压满链路，否则排队时延会改变包间隔关系。
- 轻微丢包下，`success=true` 表示接收端成功输出解码结果；`complete=false` 表示有局部符号不确定。
- 如果 `unknown_bits` 对应的原始 bit 恰好是填充值 `0`，最终文件仍可能完全匹配；这属于随机结果，不能理解为纠错能力。
