# UDP真实业务流代理测试报告

## 目标

验证策略0/1是否可以挂在真实 `iperf -u` 业务流上工作，而不是单独发策略测试包。

## 结论

可以。

在真实 UDP 业务代理模式下：

- 业务流可以连续通过代理转发，`iperf` 仍然有稳定输出。
- 策略0/1的隐蔽数据可以正常解码。
- 当“代理间带标签包”发生少量丢失时，接收端仍然能输出结果，并只在局部符号处标记不确定。

## 测试矩阵

输入隐蔽数据统一为 13 字节，即 104 bit。

### 策略0

| 用例 | 业务包 | 带标签包 | 完整性 | bit准确率 |
|---|---:|---:|---|---:|
| 无丢包 | 280 | 209/209 | complete=true | 100.00% |
| 业务先丢包 | 277 | 209/209 | complete=true | 100.00% |
| 标签包丢失(每100个丢1个) | 280 | 207/209 | complete=false | 98.08% |
| 标签包丢失(每50个丢1个) | 280 | 205/209 | complete=false | 96.15% |

### 策略1

| 用例 | 业务包 | 带标签包 | 完整性 | bit准确率 |
|---|---:|---:|---|---:|
| 无丢包 | 280 | 157/157 | complete=true | 100.00% |
| 业务先丢包 | 277 | 157/157 | complete=true | 100.00% |
| 标签包丢失(每100个丢1个) | 280 | 156/157 | complete=false | 99.04% |
| 标签包丢失(每50个丢1个) | 280 | 154/157 | complete=false | 94.23% |

## 解释

- “业务先丢包”指真实业务包在进入代理前少量缺失，这不会破坏隐蔽解码，只要后续标签包还够。
- “标签包丢失”才是真正检验方案B的场景。结果表明丢失只会让对应符号局部失效，接收端还能继续输出结果。
- 这说明方案B解决的是“丢包后整体错位”，不是恢复被丢掉的那一位。

## 阶段1一体化闭环复测（2026-06-23）

本轮验证采用当前阶段建议的闭环方式：

```text
策略/业务流：h1 -> s1 -> 固定path0 -> s2 -> h2
INT反馈流：h2 -> s2 -> 轮询path0/path1/path2 -> s1 -> h1
```

这样做的原因是：策略0/1属于时序信道，如果把同一个时序窗口直接轮询到 5/15/30ms 三条不同延迟路径，接收端看到的间隔会被多路径时延差和乱序打乱，策略1尤其容易失败。因此阶段1先证明“策略可用”和“INT可测三路径”两个能力同时存在；阶段2再根据 INT 结果为每条链路分别选择策略。

结果文件：

- `proxy-test/full_stage1_results_fixed/summary.csv`
- `proxy-test/full_stage1_results_fixed/s0/`
- `proxy-test/full_stage1_results_fixed/s1_udp_reverse/`

| 策略 | 隐蔽解码完整 | 隐蔽数据比对 | h2转发业务包 | h1 INT成功 | INT路径 | INT样本数 |

| 0 | true | 一致 | 674 | true | 0,1,2 | path0=1018, path1=496, path2=515 |
