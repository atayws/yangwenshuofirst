# UDP真实业务流代理测试说明

## 目标

这个文件夹用于验证“隐蔽策略运行在真实 UDP 业务流上”，而不是单独发送策略测试包。

当前实现文件：

```text
experiments/udp_covert_proxy.py
```

它的逻辑是：

```text
h1 iperf -u client
  -> h1 sender proxy
  -> s1 -- 多路径/INT -- s2
  -> h2 receiver proxy
  -> h2 iperf -u server
```

发送端代理从真实 UDP 业务流中取包，按策略0/1控制转发间隔，并在代理间 UDP 负载前加 2 字节方案B同步标签。接收端代理解析标签和到达时序，剥掉标签后把原始业务负载交给 iperf server。

因此 iperf server 看到的是还原后的业务包，不需要知道隐蔽传输存在。

## 推荐演示命令

先启动拓扑：

```bash
cd /home/p4/yws-covert
sudo bash run.sh
```

以下命令在 Mininet CLI 中执行。

### 1. h2 启动真实业务服务端

```bash
h2 iperf -u -s -p 5201 -i 1
```

### 2. h2 启动接收端代理

策略0使用：

```bash
h2 python3 experiments/udp_covert_proxy.py receiver --strategy 0 --listen-ip 10.0.1.2 --listen-port 6100 --forward-ip 127.0.0.1 --forward-port 5201 --expected-bytes 13 --hidden-output proxy-test/decoded_s0.bin --summary proxy-test/receiver_s0.json --timeout 40
```

策略1使用：

```bash
h2 python3 experiments/udp_covert_proxy.py receiver --strategy 1 --listen-ip 10.0.1.2 --listen-port 6101 --forward-ip 127.0.0.1 --forward-port 5201 --expected-bytes 13 --hidden-output proxy-test/decoded_s1.bin --summary proxy-test/receiver_s1.json --timeout 40
```

### 3. h1 启动发送端代理

策略0使用：

```bash
h1 python3 experiments/udp_covert_proxy.py sender --strategy 0 --hidden-input celue0/results_scheme_b/input_payload.bin --listen-ip 127.0.0.1 --listen-port 6000 --remote-ip 10.0.1.2 --remote-port 6100 --summary proxy-test/sender_s0.json
```

策略1使用：

```bash
h1 python3 experiments/udp_covert_proxy.py sender --strategy 1 --hidden-input celue1/results_scheme_b/input_payload.bin --listen-ip 127.0.0.1 --listen-port 6001 --remote-ip 10.0.1.2 --remote-port 6101 --summary proxy-test/sender_s1.json
```

### 4. h1 启动真实 UDP iperf 业务流

策略0对应发送到本地代理端口 `6000`：

```bash
h1 iperf -u -c 127.0.0.1 -p 6000 -b 300K -t 25 -i 1
```

策略1对应发送到本地代理端口 `6001`：

```bash
h1 iperf -u -c 127.0.0.1 -p 6001 -b 300K -t 20 -i 1
```

## 结果查看

接收端代理输出：

```text
proxy-test/receiver_s0.json
proxy-test/receiver_s1.json
```

重点字段：

- `packets_received`：代理间收到的总业务包数。
- `tagged_packets`：其中携带隐蔽标签的业务包数。
- `forwarded_business_packets`：还原后交给 iperf server 的业务包数。
- `success`：是否成功输出隐蔽数据。
- `decode_info.complete`：是否完整无缺失。
- `decode_info.unknown_bits` 或 `unknown_symbols`：丢包导致的不确定位置。

隐藏数据输出：

```text
proxy-test/decoded_s0.bin
proxy-test/decoded_s1.bin
```

## 注意事项

- 不要把 UDP iperf 打满链路。建议先用 `-b 300K` 或 `-b 500K`。
- 代理会按策略0/1主动调整发包间隔，所以 iperf server 看到的吞吐会低于原始发送速率，这是时序隐蔽信道的正常代价。
- P4 不解析隐蔽标签；Wireshark 可以在 h1/h2 或 s1/s2 端口看到代理间 UDP 负载前多了 2 字节标签。
- INT 可以同时开启，用来测量链路状态；隐蔽代理和 INT 是两条逻辑，互不依赖。
- 如果要模拟“隐蔽承载包在网络中丢失”，发送端代理可以加入 `--drop-tagged-every 100`。这只用于实验验证，真实 Mininet 中也可以用 `tc netem loss` 在链路上制造丢包。
