# 策略1：高容量排序时序 验证说明

## 当前设计

策略1用一个小窗口内 3 个连续间隔的排序表示 2 bit。相比固定阈值，它更关注相对排序，因此对慢变时延更稳定，容量也高于策略0。

本次采用方案B：在每个承载业务包的 UDP 负载开头加入 2 字节同步标签。标签字段包含帧编号、策略编号、相位和符号索引，并经过轻量异或混淆。标签只用于接收端重同步，不承载真实隐蔽数据；真实隐蔽数据仍然由窗口内包间隔排序承载。

丢包时，接收端会按标签判断哪个符号窗口缺了相位。如果窗口不完整，就把该符号标为 `unknown_symbols`，后续窗口仍然可以继续解码，不会整体错位。

## 文件位置

- `results_scheme_b/input_payload.bin`：104bit 测试输入数据。
- `results_scheme_b/input_bits.txt`：输入数据的二进制文本。
- `results_scheme_b/summary.csv`：离线矩阵测试汇总。
- `results_scheme_b/summary.json`：离线矩阵测试详细结果。
- `results_scheme_b/vm_mininet/`：Ubuntu 虚拟机 Mininet/BMv2 实测结果。
- `results_scheme_b/测试报告.md`：本策略测试报告。

## 手动验证流程

1. 启动拓扑：

```bash
cd /home/p4/yws-covert
sudo bash run.sh
```

2. 策略1默认使用 UDP 端口 `50001`。在 Mininet CLI 中启动接收端：

```bash
h2 python3 experiments/live_receive_strategy.py --strategy 1 --iface h2-eth0 --output celue1/results_scheme_b/manual/decoded_output.bin --summary celue1/results_scheme_b/manual/receive_summary.json --timeout 30 --expected-bytes 13
```

3. 在另一个 Mininet CLI 或 xterm 中发送承载业务流：

```bash
h1 python3 experiments/live_send_strategy.py --strategy 1 --input celue1/results_scheme_b/input_payload.bin --dst-ip 10.0.2.2 --src-ip 10.0.1.2 --send-mode socket
```

4. 如需同时验证普通业务流不断，可以开低速 UDP iperf 背景流，注意不要打满链路：

```bash
h1 iperf -u -s -p 51000
h2 iperf -u -c 10.0.1.2 -p 51000 -b 500K -t 60 -i 1
```

## 注意事项

- 不要用满速 TCP 或满速 UDP 背景流压满链路，否则排队时延会改变排序关系。
- 轻微丢包下，`success=true` 表示接收端成功输出解码结果；`complete=false` 表示有局部窗口不确定。
- 策略1每个符号承载 2 bit，所以丢一个相位包通常影响一个 2bit 符号。
