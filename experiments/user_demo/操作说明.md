# 用户演示五窗口操作说明

本目录把中期演示常用操作整理成五个窗口程序。拓扑结构固定为：

```text
h1 -- s1 == path0/path1/path2 == s2 -- h2
```

## 1. 窗口分工

| 程序 | 作用 |
|---|---|
| `topology_service.py` | 一键启动拓扑、P4、流表、后台 UDP iperf、INT 解析器，并提供本地控制接口。 |
| `sender_window.py` | 发送端窗口，输入要隐蔽传输的数据。 |
| `receiver_window.py` | 接收端窗口，实时显示 h2 解码出的隐蔽数据。 |
| `link_status_window.py` | 每 1s 读取最新 INT 状态，动态显示三条链路指标，并直接显示链路1/2/3当前建议策略。 |
| `link_config_tool.py` | 动态设置三条链路的时延和丢包率。 |

## 2. 启动顺序

先进入虚拟机项目目录：

```bash
cd /home/p4/yws-covert
```

窗口1：启动拓扑服务。这个窗口需要 `sudo`，并且不要关闭。启动前会自动清理旧 Mininet/BMv2 残留；如果上次异常退出留下 veth 接口，一般不用再手动执行 `sudo mn -c`。如果旧五窗口服务仍占用 `127.0.0.1:38765`，新服务会先请求旧服务退出再继续启动。

```bash
sudo python3 experiments/user_demo/topology_service.py --clean-results
```

看到类似下面输出后，说明服务已启动。如果终端只看到 Mininet 交换机启动和流表下发，稍等几秒；服务启动后会继续打印下面这行。

```text
[topology] 用户演示服务已启动：127.0.0.1:38765
```

窗口2：打开接收端显示。

```bash
python3 experiments/user_demo/receiver_window.py
```

窗口3：打开链路状态和建议策略看板。

```bash
python3 experiments/user_demo/link_status_window.py
```

窗口4：打开发送端窗口。

```bash
python3 experiments/user_demo/sender_window.py
```

在发送端窗口中输入任意文本，回车发送；接收端窗口会显示 h2 解码结果。

如果输入内容只包含 `0` 和 `1`，发送端会按真正的比特串处理。例如输入 `101` 时只发送 3 bit，
不会再按三个 ASCII 字符拆成 24 bit。

窗口5：按需打开链路设置工具。

```bash
python3 experiments/user_demo/link_config_tool.py
```

## 3. 设置链路时延和丢包

一次性设置：

```bash
python3 experiments/user_demo/link_config_tool.py 1 20 10 2 10 20 3 30 30
```

含义：

| 输入链路 | 实际 path | 设置 |
|---|---|---|
| `1 20 10` | path0 | delay=20ms，loss=10% |
| `2 10 20` | path1 | delay=10ms，loss=20% |
| `3 30 30` | path2 | delay=30ms，loss=30% |

也可以在交互模式中输入同样的三元组：

```text
> 1 5 0 2 15 0 3 30 0
```

恢复默认链路：

```bash
python3 experiments/user_demo/link_config_tool.py 1 5 0 2 15 0 3 30 0
```

## 4. 非交互发送

如果只想快速发一条文本：

```bash
python3 experiments/user_demo/sender_window.py --text "HELLO MIDTERM"
```

发送窗口默认使用稳定快速模式：每个 segment 开始前读取最新 INT 状态，默认在干净链路上优先使用
策略3（包长统计），链路出现明显丢包/抖动时切到策略2（IP-ID可靠）。默认不会为了“覆盖六个策略”
强行插入策略0/1/4/5，避免短消息被时序策略拖慢。

当前默认分段大小为 16 字节。早期版本曾按 6 字节切分，长输入会产生大量 segment 和路径切换，
例如 1024 字节会被切成 171 段，发送速度很慢。现在 1024 字节只切成 16 段，长输入不会再因为
过度碎片化而卡住。

策略0/1已经支持 INT 驱动的自适应时序间隔。如果需要展示时序策略，建议使用 `--cover-six`，服务会在每个
segment 开始前读取最新链路 delay/jitter：低抖动链路使用较小间隔提高速度，高抖动链路自动拉大间隔保证解码。
当前 VM 验证中，低抖动链路下策略0为 `8ms/22ms`，策略1为 `12ms/24ms/42ms`；链路抖动升高后会自动放大到
`100ms/220ms` 或 `100ms/180ms/300ms`。窗口3在策略0/1运行时会显示本次实际采用的时序参数。

如需单独展示六个策略都能挂载到业务流，可启动拓扑服务时增加：

```bash
sudo python3 experiments/user_demo/topology_service.py --clean-results --cover-six
```

## 5. 结果目录

所有结果统一写入：

```text
experiments/results/user_demo/
```

关键文件：

| 文件 | 说明 |
|---|---|
| `latest_int_summary.json` | h1 最新 INT 解析结果。 |
| `history.csv` | 每次隐蔽发送和解码历史。 |
| `summary.json` | 拓扑服务退出时写出的总摘要。 |
| `session_XXX/summary.json` | 某次发送的详细结果。 |
| `session_XXX/rule_plan.json` | 本次发送使用的策略计划。 |
| `session_XXX/proxy_plan.json` | 真实 UDP 业务流代理使用的分段、策略、路径和端口计划。 |
| `session_XXX/sender_summary.json` | h1 侧 `plan-sender` 在真实 `iperf -u` 业务流上挂载隐蔽数据的统计。 |
| `session_XXX/receiver_summary.json` | h2 侧 `plan-receiver` 识别策略、解码隐蔽数据和转发业务 payload 的统计。 |
| `session_XXX/decoded_secret.bin` | h2 解码出的隐蔽数据。 |

## 6. 常见启动问题

如果仍然看到 `RTNETLINK answers: File exists`，说明系统里还有更早的 Mininet 残留，可以手动执行：

```bash
sudo mn -c
sudo pkill -f simple_switch
```

如果提示 `Address already in use`，说明旧的五窗口拓扑服务仍在监听控制端口。优先执行：

```bash
python3 experiments/user_demo/demo_client.py shutdown
```

如果 shutdown 不生效，再手动清理：

```bash
sudo pkill -f experiments/user_demo/topology_service.py
```

## 7. 当前实现边界

当前五窗口演示适合中期审核展示：

1. h2->h1 的 UDP iperf 业务流持续运行，用于触发反向 INT。
2. h1 可以看到三条链路状态，并根据状态生成规则策略计划。
3. h1 输入隐蔽数据后，服务会启动 h1 本机 `iperf -u` 到 `plan-sender` 的真实业务流，隐蔽数据挂载在这些业务包上发往 h2。
4. 链路时延和丢包可以在运行中修改。

当前五窗口发送入口已经接入 `experiments/udp_covert_proxy.py plan-sender/plan-receiver`，并升级为
chunk 级动态重规划：h2 后台 `iperf -s -u` 接收恢复后的业务 payload，h1 用本机 `iperf -u`
持续喂给发送代理；每个 segment 开始前，拓扑服务重新读取最新 INT 状态，为该 segment 选择策略和路径，
改写 `proxy_plan.json`，再放行真实业务包。接收代理会动态重新加载计划，解码后再把原始 payload
转发给 h2 的 `iperf` server。

窗口3保留 INT 表格，同时把策略显示简化为：

```text
链路1: 策略3 包长统计
链路2: 策略3 包长统计
链路3: 策略3 包长统计
```

如果运行中设置链路1丢包，例如：

```bash
python3 experiments/user_demo/link_config_tool.py 1 5 5 2 15 0 3 30 0
```

链路状态窗口会显示链路1建议切换为策略2：

```text
链路1: 策略2 IP-ID可靠
链路2: 策略3 包长统计
链路3: 策略3 包长统计
```

发送端实际选策略时会同时参考 INT 测量值和窗口5设置的 netem 参数；如果某条链路刚被设置为
丢包链路，后续 segment 会立即避开该链路，或者在无健康链路可用时切到策略2并提高重复强度。

VM 最新验证结果：

| 输入 | 结果 | 耗时 | 说明 |
|---|---|---:|---|
| 64 字节文本 | `success=true`，误码率 `0%` | 约 11.4s | 1 个 segment，策略3，真实 UDP 业务流承载。 |
| 256 字节文本 | `success=true`，误码率 `0%` | 约 11.8s | 4 个 segment，默认链路下走策略3。 |
| 1024 字节文本 | `success=true`，误码率 `0%` | 约 28.0s | 16 个 segment，较旧版约 87.8s 明显提速。 |
| path0 设置 2% 丢包后发送 256 字节 | `success=true`，误码率 `0%` | 约 12.0s | 自动避开丢包链路，只走 path1/path2。 |
| 96 bit 覆盖策略0/1 | `success=true`，误码率 `0%` | 约 19.6s | `--cover-six` 下前两个 segment 分别走策略0和策略1，确认自适应时序配置生效。 |

当前切换粒度是 chunk/segment 边界，不是包级任意切换；这样可以避免策略0/1时序窗口、
策略4喷泉码窗口和策略5路径序列窗口在包中途被切坏。
