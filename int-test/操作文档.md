# INT 功能检测操作文档

这个文件夹专门用于查看和复现实验阶段 1 的 INT 验证结果。

## 1. 目录内容

- `操作文档.md`：本操作说明。
- `INT测试报告.md`：已经跑过的一轮测试报告。
- `results/summary.csv`：测试汇总表，适合直接导入 Excel。
- `results/summary.json`：测试汇总 JSON。
- `results/*.json`：每个 case 的 h1 INT 解析结果。
- `results/*_iperf.txt`：每个 case 的 iperf 业务流输出。

## 2. 当前 INT 实现原理

当前拓扑：

```text
h1 -- s1 == 三条链路 == s2 -- h2
```

三条交换机间链路：

```text
path0: s1-eth2 <-> s2-eth2，默认 5ms
path1: s1-eth3 <-> s2-eth3，默认 15ms
path2: s1-eth4 <-> s2-eth4，默认 30ms
```

当前阶段验证的是 h2 到 h1 方向的业务流：

```text
h2 -> s2 -> 三条链路之一 -> s1 -> h1
```

INT 现在直接插入采样命中的真实业务包，但只在 s2-s1 交换机间链路存在：

1. h2 正常向 h1 发送业务流，例如 iperf。
2. s2 按采样间隔从业务流中选包。
3. s2 在这个真实业务包的 IPv4 头后插入 INT shim 和第一跳 probe_data。
4. 携带 INT 的业务包从 s2 经过实际选择的链路到 s1。
5. s1 补齐本端 probe_data 后本地复制两份：一份剥离 INT 并恢复原业务包交给 h1，另一份转换成 UDP/50100 INT 报告交给 h1。
6. h1 上的 Python 程序监听 UDP/50100，解析出每条链路的时延、丢包、带宽相对负载和队列深度。

这样做的好处是：INT 真正运行在业务包上，但 h1/h2 看到的业务流仍然是正常 TCP/UDP/ICMP，不需要知道中间是否做了 INT。

`run.sh` 会把终端侧 MTU 保持为 1500，把 s1-s2 三条交换机间链路 MTU 设置为 1600，用于容纳业务包临时增加的 INT shim 和两跳 probe_data。

## 3. 一次最小验证流程

进入虚拟机项目目录：

```bash
cd /home/p4/yws-covert
```

启动拓扑：

```bash
sudo bash run.sh
```

进入 Mininet CLI 后，先检查基础连通性：

```bash
h1 ping -c 5 10.0.1.2
h2 ping -c 5 10.0.1.1
```

在 h1 启动 INT 接收端：

```bash
h1 python3 experiments/reverse_probe_receiver.py \
  --timeout 30 \
  --window-ms 30000 \
  --output experiments/results/manual_int_check.json &
```

启动业务流：

```bash
h1 iperf -s -i 5 &
h2 iperf -c 10.0.1.1 -t 20 -i 5
```

等待 receiver 结束后查看结果：

```bash
h1 cat experiments/results/manual_int_check.json
```

重点看这些字段：

- `parsed_int_reports`：h1 成功解析的 INT 报告数量。
- `metric_sample_counts`：每条路径的 INT 指标样本数。
- `path_states`：每条路径的链路状态。
- `delay_ms`：实测时延。
- `loss_rate`：实测丢包率。
- `bw_utilization`：相对带宽负载，阶段 1 用于趋势判断。

## 4. 切换单路路径

打开另一个终端，进入 s2 的 simple_switch_CLI：


固定走 path0：

```text
register_write reg_path_mode 0 1
register_write reg_fixed_path 0 0
register_write reg_int_interval_us 0 10000
register_write reg_next_sample_time 0 0
```

```text
register_write reg_path_mode 0 1
register_write reg_fixed_path 0 1
```

固定走 path2：

## 5. 切换多路轮询

s2：

```bash
simple_switch_CLI --thrift-port 9091
```

```text
register_write reg_path_mode 0 2
register_write reg_rr_burst_size 0 12
register_write reg_rr_counter 0 0
register_write reg_rr_current_path 0 0
register_write reg_int_interval_us 0 10000
register_write reg_next_sample_time 0 0
register_write reg_int_enabled 0 1
```

s1：

```bash
simple_switch_CLI --thrift-port 9090
```

```text
register_write reg_path_mode 0 2
register_write reg_rr_burst_size 0 12
register_write reg_rr_counter 0 0
register_write reg_rr_current_path 0 0
register_write reg_int_enabled 0 0
```

这里 `reg_rr_burst_size = 12` 表示每条路径连续转发 12 个包后切到下一条路径。这个值后续可以根据隐蔽策略窗口长度调整。

## 6. 设置链路时延和丢包

在 Mininet CLI 中可以用 `tc` 修改链路条件。

设置 path0 为 20ms、0% 丢包：

```bash
s2 tc qdisc replace dev s2-eth2 root netem delay 20ms loss 0%
```

设置 path1 为 15ms、10% 丢包：

```bash
s2 tc qdisc replace dev s2-eth3 root netem delay 15ms loss 10%
```

设置 path2 为 55ms、0% 丢包：

```bash
s2 tc qdisc replace dev s2-eth4 root netem delay 55ms loss 0%
```

恢复默认三条链路：

```bash
s2 tc qdisc replace dev s2-eth2 root netem delay 5ms loss 0%
s2 tc qdisc replace dev s2-eth3 root netem delay 15ms loss 0%
s2 tc qdisc replace dev s2-eth4 root netem delay 30ms loss 0%
```

## 7. 本次已经完成的测试

本次测试采样间隔为 10ms，每个 case 的 iperf 持续 12s。

已经测试的 case：

- `single_p0_base`：path0，5ms，0% 丢包。
- `single_p0_delay20`：path0，20ms，0% 丢包。
- `single_p0_loss10`：path0，5ms，10% 丢包。
- `single_p1_base`：path1，15ms，0% 丢包。
- `single_p1_delay35`：path1，35ms，0% 丢包。
- `single_p1_loss10`：path1，15ms，10% 丢包。
- `single_p2_base`：path2，30ms，0% 丢包。
- `single_p2_delay55`：path2，55ms，0% 丢包。
- `single_p2_loss10`：path2，30ms，10% 丢包。
- `multi_base`：三路径轮询，默认 5/15/30ms，0% 丢包。
- `multi_p1_loss10`：三路径轮询，path1 设置 10% 丢包。
- `multi_delay_skew`：三路径轮询，三条链路设置为 10/30/60ms。

结论摘要：

- `iperf` 业务流 18/18 条记录都正常完成。
- INT 18/18 条链路记录都有样本。
- 时延测量能稳定区分三条路径，实测值约为配置时延加 1ms 到 2.7ms 的 BMv2/Mininet 开销。
- 丢包测量能反映趋势，但 12s 短窗口下会有采样误差；如果用于论文表格，建议再跑 30s 或 60s 长窗口。

## 8. 如何看本文件夹里的结果

优先打开：

```text
INT测试报告.md
results/summary.csv
```

如果要看某一组的详细原始数据，例如多路径基线：

```text
results/multi_base.json
results/multi_base_iperf.txt
```

`*.json` 里：

- `int_reports` 保存前几个原始 INT 报告样例。
- `raw_metrics` 保存最近一次计算得到的原始指标。
- `path_states` 保存滑动窗口聚合后的链路状态。

`*_iperf.txt` 里可以看到该 case 的 TCP 业务流是否正常完成。
