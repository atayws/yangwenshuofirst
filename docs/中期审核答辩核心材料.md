# 中期审核答辩核心材料

## 1. 一句话说明项目

本课题面向低空网络中“地面端 h1/s1 到低空端 s2/h2”的多路径通信场景，设计一个在正常业务流不中断的前提下，利用业务包时序、IP-ID、包长和路径序列等特征传输隐蔽数据的原型系统。系统通过 P4 交换机完成多路径转发和 INT 链路状态采集，通过终端侧 Python 策略库完成隐蔽数据编码、接收识别、解码和按序重组。

当前中期阶段已经实现的是：不用 PPO，先用规则选择器代替强化学习；在 `h1--s1--三条链路--s2--h2` 拓扑中，UDP iperf 作为真实业务流持续运行，隐蔽数据挂载在这些业务包上，接收端解码后继续把原始业务 payload 转发给 iperf server，业务流不中断。

需要准确说明的一点：当前六个策略不是再额外创造一条独立“策略包流”，而是在终端侧 UDP 业务流代理中使用真实 UDP 业务包作为承载。实现上是 h1 本机 `iperf -u` 先把业务包发给发送代理，发送代理在这些业务 payload、IP-ID、包长、发送时间或路径选择上叠加隐蔽特征；h2 接收代理抓到承载包后识别和解码隐蔽数据，再剥离代理层小头或忽略 IP-ID 标记，把原始业务 payload 转发给 h2 本机 `iperf -s -u`。

## 2. 总体数据流

### 2.1 网络拓扑

```text
h1 -- s1 == path0 / path1 / path2 == s2 -- h2
```

- h1：地面端，输入隐蔽数据，接收 INT 链路状态，运行发送代理。
- s1：地面端 P4 交换机，负责 h1->h2 多路径转发，也负责接收反向 INT 并剥离 INT。
- s2：低空端 P4 交换机，负责 h2->h1 方向的 INT 插入和 h1->h2 业务转发。
- h2：低空端，运行接收代理和业务 server，解码隐蔽数据。
- path0/path1/path2：三条可设置 delay/loss 的链路，用于模拟低空网络链路差异。

### 2.2 INT 和隐蔽传输的方向

当前中期演示中，两条业务方向分工如下：

```text
h2 -> h1：后台 UDP iperf，用于触发反向 inline INT。
h1 -> h2：前向 UDP iperf，用于承载隐蔽数据。
```

h2->h1 方向持续发 UDP 业务流，s2 按采样间隔从真实业务包中选一个包，在 IPv4 头后插入 compact INT shim 和 probe_data。该 INT 头只在 s2-s1 之间存在。s1 收到后补齐本端信息，生成本地 UDP/50100 INT 报告给 h1 解析程序，同时剥离 INT 头并恢复原业务包交给 h1。所以普通应用不会收到带 INT 的包。

h1->h2 方向由用户输入隐蔽数据，发送代理把隐蔽信息挂载到真实 UDP 业务包上，经 s1 多路径转发到 s2/h2。h2 接收代理识别策略并解码，同时把原始业务 payload 转发给 iperf server。

### 2.3 当前业务流代理的真实含义

当前不是直接修改 Linux 内核里已经发出的 iperf 原包，而是在应用层代理中完成“接收原始业务 payload -> 叠加隐蔽特征 -> 重新发出 UDP 包”。这仍然是真实业务流承载，因为发送端代理使用的是 iperf 产生的业务 payload 和业务节奏，接收端也会把还原后的 payload 交给 iperf server，业务链路可通过 iperf 连续性验证。

代理处理逻辑在：

```text
experiments/udp_covert_proxy.py

extract_business_payload()  # 接收端剥离小头，还原业务 payload
run_plan_sender()           # 多 segment 发送代理
run_plan_receiver()         # 多策略接收代理
```

## 3. INT 如何提供链路状态

### 3.1 INT 头部插入位置

当前 INT 采用 inline 方式：

```text
Ethernet | IPv4(protocol=0xFD) | INT shim | probe_data[] | 原 UDP/TCP/ICMP 负载
```

其中 INT shim 已压缩为 4 字节：

```text
byte0: version(2 bit) + flags(2 bit) + hop_count(4 bit)
byte1: original_protocol(8 bit)
byte2-3: trace_id(16 bit)
```

- `version`：协议版本，当前用于后续兼容。
- `flags`：预留标志位，普通 inline INT 使用 0。
- `hop_count`：当前已经写入的 probe_data 数量。
- `original_protocol`：保存原 IPv4 protocol，例如 UDP=17。s1 剥离 INT 后用它恢复 IPv4 protocol。
- `trace_id`：同一路径的采样序号，用于检查 INT 样本连续性，不直接作为业务链路丢包率。

probe_data 记录交换机观测点信息，例如 ingress/egress 端口、时间戳、累计字节、累计包数和队列深度。当前两交换机情况下，主要用 s2 出口时间戳与 s1 入口时间戳估计 s2-s1 链路时延。

### 3.2 链路状态计算

h1 端 INT 解析程序读取 UDP/50100 报告，按路径统计：

- `delay_ms`：s1 ingress timestamp - s2 egress timestamp，近似交换机间链路时延。
- `jitter_ms`：相邻 INT 样本 delay 的变化。
- `loss_rate`：用连续两份 INT 中“源交换机出端口累计包数差”和“下一跳/终点交换机入端口累计包数差”计算，即 `(sent_delta - recv_delta) / sent_delta`，表示交换机间链路段的业务包丢包趋势。
- `sample_loss_rate`：由 `trace_id` 跳变估计的 INT 样本缺口，只作为采样连续性诊断，不作为策略选择的主丢包指标。
- `bw_utilization`：端口累计字节差和采样间隔估算的负载趋势。
- `qdepth`：BMv2 队列深度，用于观察拥塞趋势。

## 4. 六个策略是否都挂载在真实业务包上

是。当前六个策略的中期验证入口都接入 `experiments/udp_covert_proxy.py`。区别只是每个策略修改业务包的哪个外在特征：

| 策略 | 承载位置 | 是否改变原业务 payload | 接收端如何还原业务 payload |
|---|---|---|---|
| 策略0 | 业务包发送间隔 + UDP payload 前 2 字节同步标签 | 前面临时加 2 字节同步标签 | 解码后去掉前 2 字节再转发 |
| 策略1 | 业务包发送间隔 + UDP payload 前 2 字节同步标签 | 前面临时加 2 字节同步标签 | 解码后去掉前 2 字节再转发 |
| 策略2 | IPv4 Identification 字段 | 不改业务 payload | 直接转发原 payload |
| 策略3 | IP 包长区间 + UDP payload 前 12 字节同步小头 + 2 字节原 payload 长度 | 前面临时加小头和长度字段，并补 padding 控制包长 | 按长度字段截出原 payload 再转发 |
| 策略4 | IPv4 Identification 字段 | 不改业务 payload | 直接转发原 payload |
| 策略5 | IP-ID 自描述字段 + 路径序列/路径提示 | 不改业务 payload | 直接转发原 payload |

这里的“临时加小头”只发生在代理间承载包中，接收端代理会剥离，不会交给业务应用。

## 5. 策略0：高隐蔽滑动二阶差分时序

代码路径：

```text
python/covert_strategies/timing_high_covert.py
python/covert_strategies/timing_sync_tag.py
experiments/udp_covert_proxy.py
```

### 5.1 核心思想

策略0不把真实隐蔽比特写进明显字段，而是通过业务包到达时间间隔承载信息。连续 4 个带标签业务包形成一个窗口：

```text
P0, P1, P2, P3

d1 = t(P1) - t(P0)
d2 = t(P2) - t(P1)
d3 = t(P3) - t(P2)

score = (d1 - d2) - (d2 - d3) = d1 - 2*d2 + d3
```

判决规则：

```text
score < -T  -> bit 0
score > +T  -> bit 1
abs(score) <= T -> 不可靠，标记 unknown
```

窗口采用滑动复用：第一个窗口用 P0-P3，第二个窗口丢掉 P0、加入 P4，用 P1-P4。这样相邻 bit 复用 3 个包，比旧版“两个间隔传一个 bit”效率更高。

### 5.2 怎么挂载到业务包

发送端代理收到 iperf 业务 payload 后，并不是新造一条独立策略流，而是从策略0的 `encode()` 得到一组 `PacketSpec`。每个 `PacketSpec` 给出：

- 该业务包前面要加的 2 字节同步标签；
- 相对上一个承载包应等待的发送间隔 `send_delay_ms`；
- 该包所在路径 `path_id`；
- 当前策略编号 `strategy_id=0`。

发送代理处理时：

```text
原业务 payload -> 2字节 timing tag + 原业务 payload -> 按 send_delay_ms 发出
```

2 字节标签只用于同步，不承载真实隐蔽 bit。真实 bit 来自包间隔的二阶差分。

### 5.3 什么时候插入、什么时候解码

插入时机：只有用户在发送窗口输入隐蔽数据，并且当前 segment 被策略选择器分配给策略0时，发送代理才对后续若干个真实业务包叠加策略0标签和时序控制。没有隐蔽数据时，业务包按普通 UDP 业务流转发。

解码时机：h2 接收代理抓到对应 UDP 端口的业务包后，把 payload 和 arrival_time_ms 交给统一接收分发器。分发器看到 payload 前 2 字节能解析出策略0 timing tag，就放入策略0缓冲区。该 segment 结束后调用策略0 `decode()`，用到达时间计算 d1/d2/d3 和 score，恢复 bit 流。

### 5.4 怎么恢复原业务流

接收代理识别到策略0承载包后：

```text
2字节 timing tag + 原业务 payload -> 去掉前2字节 -> 原业务 payload -> 转发给 iperf server
```

所以 iperf server 看到的是原始业务 payload，不会看到同步标签。

### 5.5 自适应阈值如何来自 INT

策略0的自适应配置在：

```text
experiments/user_demo/topology_service.py
timing_config_for_state(strategy_id=0, state)
```

输入 `state` 来自 INT 解析出的链路状态，主要用 `delay_ms` 和 `jitter_ms`。当前规则是：

```text
threshold = clamp(max(2.0, jitter_ms*3.0, delay_ms*0.015), 2.0, 35.0)
base_gap  = clamp(max(6.0, jitter_ms*4.0, delay_ms*0.04), 6.0, 80.0)
delta     = clamp(max(threshold*2.4, jitter_ms*6.0, delay_ms*0.04, 6.0), 6.0, 90.0)
sliding_window_block_symbols = 4
```

解释口径：链路 jitter 越大，阈值 T 越大，避免普通抖动被误判成隐蔽 bit；delay 越大或 jitter 越大，基础间隔也适当增大，提高解码稳定性。低时延低抖动链路下，base_gap 可以降到 6ms，提高速率。

### 5.6 适用场景和限制

策略0适合低丢包、低抖动链路，隐蔽性较强，但容量低。因为它依赖时间窗口，同一个策略0 segment 不适合在三条不同 delay 的路径上轮询，否则路径差异会破坏包间隔关系。因此策略0通常绑定单路径。

## 6. 策略1：高容量四档滑动二阶差分时序

代码路径：

```text
python/covert_strategies/timing_high_capacity.py
python/covert_strategies/timing_sync_tag.py
experiments/udp_covert_proxy.py
```

### 6.1 核心思想

策略1和策略0使用相同的 4 包滑动窗口，也计算：

```text
score = (d1 - d2) - (d2 - d3)
```

区别是策略1用四个区间表示 2 bit：

```text
score < -T2          -> 00
-T2 <= score < -T1   -> 01
+T1 < score <= +T2   -> 10
score > +T2          -> 11
-T1 <= score <= +T1  -> 不可靠，标记 unknown
```

所以策略1容量约为策略0的 2 倍，但对时延抖动更敏感。

### 6.2 怎么挂载到业务包

和策略0一样，发送端代理对真实业务 payload 前加 2 字节 timing tag，并按 `send_delay_ms` 控制承载包发送间隔。标签用于告诉接收端：这是策略1、第几个窗口、第几个包；真实 2 bit 仍由四档 score 承载。

### 6.3 自适应阈值

配置函数：

```text
experiments/user_demo/topology_service.py
timing_config_for_state(strategy_id=1, state)
```

当前规则：

```text
threshold1 = clamp(max(2.0, jitter_ms*2.5, delay_ms*0.012), 2.0, 25.0)
threshold2 = clamp(max(threshold1*2.8, jitter_ms*5.0, delay_ms*0.03), threshold1*2.2, 70.0)
score_step = clamp(max(threshold1*1.8, jitter_ms*4.0, delay_ms*0.025, 5.0), threshold1*1.2, threshold2)
base_gap   = clamp(max(6.0, jitter_ms*3.5, delay_ms*0.035), 6.0, 70.0)
sliding_window_block_symbols = 4
```

解释口径：T1 是内层保护带，T2 是四档分界线。链路越抖，T1/T2 越大，四档之间距离越远；链路质量好时，间隔和阈值会变小，提高传输速率。

### 6.4 解码和业务恢复

接收端与策略0相同：分发器解析 2 字节 timing tag，按 `(strategy_id, message_key)` 分组，调用策略1 `decode()`。业务 payload 转发时去掉前 2 字节标签。

## 7. 策略2：IP-ID 高可靠存储型

代码路径：

```text
python/covert_strategies/protocol_high_reliability.py
experiments/udp_covert_proxy.py
python/receiver/strategy_router.py
```

### 7.1 核心思想

策略2把隐蔽信息写入 IPv4 Identification 字段。IP-ID 只有 16 bit，所以不能直接放太多数据。当前布局是：

```text
bit15       covert_flag = 1
bits14-12   strategy_id = 2
bits11-8    seq_mod，块内片段序号 0~15
bits7-0     encrypted_value
```

`encrypted_value` 解密后分为：

```text
高4 bit: block_mod，块号模16，用于乱序后分组
低4 bit: data_nibble，真实数据半字节或冗余半字节
```

每个块包含：

```text
12 个数据半字节 + 3 个 XOR 冗余半字节 + 1 个认证半字节 = 16 个片段
```

默认每个片段重复发送多次。这样可以抗轻微丢包，并通过认证半字节降低普通 IP-ID 误判概率。

### 7.2 怎么挂载到业务包

发送代理收到真实业务 payload 后，策略2只改 IPv4 ID：

```text
原业务 payload 不变
IPv4 ID = flag + strategy_id + seq_mod + encrypted_value
```

也就是说，业务 payload 完全不加小头，不需要剥离。接收端转发业务 payload 时直接原样转发给 iperf server。

### 7.3 怎么区分和排序

接收端分发器检查 IP-ID：

```text
bit15 是否为 1
strategy_id 是否为 2
seq_mod 是否在 0~15
解密后 block_mod 是否合理
```

同一块内根据 `seq_mod` 放回正确位置，多路径乱序不会影响块内顺序。块号用 `block_mod` 支持中期测试规模内的块重组。认证半字节用于判断整块是否可信。

### 7.4 适用场景

策略2容量较低，但鲁棒性高，适合丢包率较高或时延抖动较大的链路。当前规则选择器在链路不适合策略3时，会优先切到策略2。

## 8. 策略3：包长统计分布型

代码路径：

```text
python/covert_strategies/statistical_fusion.py
experiments/udp_covert_proxy.py
python/receiver/strategy_router.py
```

### 8.1 核心思想

策略3通过 IP 包长区间表示隐蔽符号。当前用 4 个包长区间承载 2 bit，例如某个包长度落在区间0/1/2/3，对应 00/01/10/11。为了避免包长模式太死板，策略3加入：

- 数据白化，避免原始 bit 分布过于集中；
- 加密同步小头，标记 seq_num、symbol_index、total_symbols、repeat_index；
- 重复投票，提高抗丢包能力；
- 区间内伪随机选具体长度，避免每个符号总是固定长度；
- 分类 margin，允许实际长度有轻微偏差。

### 8.2 怎么挂载到业务包

发送代理会在真实业务 payload 前加入策略3小头，并记录原始业务 payload 长度：

```text
12字节策略3同步小头 | 2字节原payload长度 | 原业务payload | padding
```

padding 的长度由目标 IP 包长决定，使最终 IP 包长度落入对应符号的长度区间。

### 8.3 接收端如何解码和恢复业务

接收端分发器先解析策略3同步小头，确认这是策略3承载包，并读取 packet_length。然后根据包长落入哪个区间恢复 2 bit 符号，多次重复的同一 symbol 用投票恢复。

业务恢复时：

```text
跳过12字节策略3小头
读取2字节原payload长度
截取原业务payload
丢弃后面的padding
转发原业务payload给iperf server
```

### 8.4 适用场景

策略3是当前演示里的默认快速策略，适合低丢包、低抖动、业务 payload 不接近 MTU 的链路。优点是容量比策略0/1高，且不依赖精细时序；限制是会改变包长分布，如果业务本身包长非常固定或接近 MTU，需要调整 iperf `-l` 或 MTU。

## 9. 策略4：IP-ID 喷泉码多路径协同

代码路径：

```text
python/covert_strategies/full_path_redundancy.py
experiments/udp_covert_proxy.py
experiments/run_manual_policy_live.py
p4/covert_int_switch.p4
```

### 9.1 核心思想

策略4用于多路径协同，不能只单路径单独出现。它把隐蔽数据拆成 frame，每个 frame 生成多个喷泉码符号。每个符号写入 IPv4 ID：

```text
bit15       flag = 1
bits14-12   strategy_id = 4
bits11-8    frame_id
bits7-4     symbol_id
bits3-0     encrypted coded_nibble
```

发送端生成的符号数量 `num_output` 大于源符号数 `k`。接收端只要在一个 frame 中收到足够多线性独立符号，就可以通过 GF(2) 消元恢复原始半字节。因此它天然适合丢包和多路径乱序场景。

### 9.2 怎么挂载到业务包

策略4只修改 IP-ID，不改变业务 payload。发送代理把每个喷泉码符号写入当前业务包 IPv4 ID，再由 P4 按加权轮询分发到两条或三条路径。业务 payload 原样转发。

### 9.3 多路径如何配置

当某个 segment 选择策略4时，控制面会调用：

```text
experiments/run_manual_policy_live.py
configure_s1_for_entry(strategy_id=4, paths)
```

P4 侧设置为加权轮询模式：

```text
reg_path_mode = PATH_MODE_WEIGHTED_RR
reg_weight_path0/1/2 = 对应路径权重
```

所以策略4的包会被分散到多个路径上。接收端不需要知道每个符号从哪条链路来，只要按 IP-ID 的 frame_id 和 symbol_id 收集即可。

### 9.4 解码与业务恢复

接收端分发器解析 IP-ID，识别策略4、frame_id、symbol_id 和 coded_nibble。策略4解码器按 frame 收集符号，收到足够符号后解码，并通过帧头 magic、长度和 CRC 校验最终 payload。业务 payload 没有小头，直接转发给 iperf server。

### 9.5 适用场景

策略4适合三条链路都不稳定或需要多路径协同时使用。优点是抗丢包、抗乱序；缺点是 IP-ID 承载容量小，有编码冗余，单次 frame 数受 4 bit frame_id 限制，长数据需要由上层 segment 切分。

## 10. 策略5：路径序列隐蔽信道

代码路径：

```text
python/covert_strategies/path_sequence.py
experiments/udp_covert_proxy.py
experiments/run_manual_policy_live.py
p4/covert_int_switch.p4
```

### 10.1 核心思想

策略5用一个窗口内的三包路径排列表示 2 bit。默认映射：

```text
0 -> path0,path1,path2
1 -> path0,path2,path1
2 -> path1,path0,path2
3 -> path1,path2,path0
```

同时，为了让接收端能区分策略和恢复窗口顺序，IP-ID 中写入轻量自描述字段：

```text
bit15       flag = 1
bits14-12   strategy_id = 5
bits11-10   path_id
bits9-0      encrypted fragment_id_mod
```

这里真实隐蔽数据来自路径序列，IP-ID 主要用于识别策略、提示路径号和片段顺序，不直接放真实 payload 数据。

### 10.2 怎么挂载到业务包

发送端为每个 2 bit 符号生成 3 个业务包，每个包给出期望 path_id，并在 IP-ID 写入策略号、路径号和加密片段号。P4 对策略5使用 IP-ID hint 模式，从 IP-ID 中读取 path_id，把包送到对应路径。

### 10.3 接收端如何解码

接收端分发器通过 IP-ID 识别策略5，并恢复 path_id 与 fragment_id_mod。策略5解码器把每 3 个 slot 组成一个窗口，根据窗口内 path_id 排列查表恢复 2 bit。数据帧包含 magic、长度和 CRC，避免错误路径排列被误当成有效结果。

### 10.4 业务恢复与限制

策略5不改业务 payload，接收端直接转发原 payload。它适合低丢包、路径差异明显且多路径可控的场景。限制是每 2 bit 需要 3 个包，若丢包会导致窗口不完整，所以代理层支持重复挂载来提高轻微丢包下的可解码概率。

## 11. 数据切分：chunk 和 segment 是什么

### 11.1 chunk

chunk 是全局隐蔽数据重组层的概念。发送端会把用户输入的长隐蔽数据切成若干块，每块带全局头：

```text
magic/version/session_id/chunk_id/total_chunks/payload_len/header_len/crc32/payload
```

这样做的目的是：不同 chunk 可以交给不同策略、不同路径发送；接收端即使乱序收到，也可以按 `chunk_id` 重组原始隐蔽数据。

相关代码：

```text
python/covert_strategies/session.py
CovertSessionFramer
CovertSessionAssembler
```

### 11.2 segment

segment 是当前真实业务流代理计划里的执行单元。中期五窗口演示中，一个 segment 基本对应一个隐蔽数据块和一次策略选择。默认分段大小：

```text
experiments/user_demo/topology_service.py
--chunk-size 默认 64 字节
```

流程是：

```text
用户输入隐蔽数据
-> 按 64 字节左右切成多个 segment
-> 每个 segment 开始前读取最新 INT path_states
-> 规则选择器选择该 segment 的策略和路径
-> 写入 proxy_plan.json
-> plan-sender 放行该 segment 的业务包挂载
-> plan-receiver 解码该 segment
-> 最后按 segment_id 拼回完整隐蔽数据
```

为什么不是每个包都切换策略：策略0/1需要完整时序窗口，策略4需要完整喷泉码 frame，策略5需要完整路径序列窗口。如果包级任意切换，会破坏这些策略内部结构。因此当前中期实现采用 chunk/segment 边界切换。

## 12. 接收端怎么区分哪个策略

统一接收分发器在：

```text
python/receiver/strategy_router.py
StrategyReceiverRouter
```

每个收到的业务包会被转换成：

```text
payload + metadata(ip_id, packet_length, arrival_time_ms, dport, message_key, path_id等)
```

识别优先级：

1. 控制面显式指定 `force_strategy_id`。五窗口 plan-receiver 知道每个 segment 对应的端口和策略，会把该 segment 的包强制归到正确策略，降低误判。
2. IP-ID 识别策略2/4/5。检查 flag、strategy_id、frame_id、symbol_id、seq_mod 等字段。
3. payload 小头识别策略3。解析 12 字节加密同步小头和认证。
4. payload 前 2 字节识别策略0/1。解析 timing tag。
5. 其他普通业务包忽略，不进入隐蔽解码缓冲区，但仍会作为普通业务 payload 转发。

每个策略的包会按 `(strategy_id, message_key)` 分组。`message_key` 通常来自 segment 的 `sequence_num` 或 tag 中的 frame_id，用于区分同一策略的不同 segment。

## 13. 乱序问题怎么解决

系统分三层解决乱序：

### 13.1 策略内部乱序

- 策略0/1：2 字节 timing tag 记录 packet_index，接收端按 packet_index 建立窗口，而不是简单按抓包顺序解码。
- 策略2：IP-ID 中的 `seq_mod` 和解密后的 `block_mod` 用于块内重排，XOR 冗余可恢复部分丢失。
- 策略3：同步小头中有 `symbol_index`、`total_symbols`、`repeat_index`，接收端按 symbol_index 投票。
- 策略4：IP-ID 中有 `frame_id` 和 `symbol_id`，接收端按 frame 收集喷泉码符号，顺序不重要。
- 策略5：IP-ID 中有 `fragment_id_mod` 和 path_id，接收端按窗口/slot 重排路径序列。

### 13.2 多策略 segment 乱序

不同 segment 由不同 UDP remote_port 和 `sequence_num` 区分，分发器按 `(strategy_id, message_key)` 建立缓冲区，不会把不同策略或不同 segment 混在一起。

### 13.3 全局消息顺序

所有 segment 最终按 `segment_id` 拼接。全局 chunk 模式下则按 `chunk_id` 拼接。每个 chunk 有 CRC，错误 chunk 会被拒绝。

## 14. 策略选择和动态切换

当前中期阶段不用 PPO，使用规则选择器：

```text
python/control_plane/rule_policy_selector.py
experiments/user_demo/topology_service.py
```

基础规则：

- 低丢包、低抖动、低时延链路：优先策略3，因为容量高、速度快。
- 链路丢包或抖动较高：优先策略2，因为 IP-ID 分块冗余更可靠。
- 至少两条路径可用时：可加入策略4喷泉码多路径协同。
- 需要展示六个策略时：开启 `--cover-six`，前几个 segment 强制覆盖策略0/1/3/4/5/2。
- 策略0/1只在低丢包低抖动链路上展示，不作为长数据默认策略。

五窗口演示的在线切换粒度是 segment 级：每个 segment 开始前重新读取最新 INT 状态。如果传输很长的数据，传输过程中修改链路 delay/loss，后续 segment 会按新状态重新选择策略和路径。

## 16. 答辩时建议强调的边界

1. 当前 P4 交换机不解析隐蔽数据，只负责转发、多路径调度和 INT。隐蔽策略全部在终端 Python 层实现。
2. 当前真实业务承载通过 UDP 代理实现，业务 payload 来自 iperf，接收端会还原并转发给 iperf server；不是额外单独发策略测试包。
3. INT 是真实业务包 inline 插入，不是单独探测流；INT 头只在 s1-s2 之间存在，终端应用无感。
4. 当前策略切换是 segment 级，不是包级。这样能保护时序窗口、喷泉码 frame 和路径序列窗口不被打断。
5. PPO 是下一阶段工作；当前规则选择器已经预留同样的策略计划接口，后续 PPO 只需要输出同格式 `PolicyEntry`。
