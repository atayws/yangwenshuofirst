# 策略5测试报告

## 1. 测试目标

验证策略5“IP-ID 自描述 + 多路径路径序列隐蔽信道”是否能在策略库层完成编码、乱序重组和局部缺包处理。策略5的真实隐蔽数据不直接写入 IP-ID，而是由三包窗口内的路径排列承载；IP-ID 只保存策略号、路径号和加密片段号，帮助接收端识别策略和解决顺序问题。

## 2. 当前编码设计

路径排列承载真实隐蔽数据：

```text
00 -> path0, path1, path2
01 -> path0, path2, path1
10 -> path1, path0, path2
11 -> path1, path2, path0
```

IP-ID 自描述字段：

```text
bit15      valid = 1
bits14-12 strategy_id = 5
bits11-10 path_id
bits9-0   encrypted fragment_id_mod1024
```

其中 `fragment_id_mod1024` 只用于排序和窗口重组，不承载真实隐蔽数据；10 bit 顺序空间可覆盖 1024 个承载包。

## 3. 测试数据

- 随机种子：`20260626`。
- 输入长度：`100` bit，落盘为 `/home/p4/yws-covert/celue5/input_payload.bin`。
- 生成承载包：`252` 个。
- 每个窗口承载：`2` bit，窗口大小 `3` 个包。

## 5. 结论

- 理想顺序和全局乱序场景均能完整恢复，说明 IP-ID 中的加密片段号和 `fragment_id` 元数据都能解决接收乱序问题。
- 数据区丢 1 个包时只造成 1 个窗口 unknown，本次 100 bit 中 99 bit 匹配；数据区稀疏丢 3 个包时 97 bit 匹配，后续窗口没有整体错位。
- 一个数据窗口路径排列异常时只造成局部 unknown，表现和单个窗口缺包类似。
- 帧头区丢包会导致 magic 校验失败并拒绝输出完整明文，这是当前策略5的主要边界；如果后续要提高抗丢包能力，可以给帧头窗口做极少量重复或把帧头信息放入策略2/4的可靠字段。
- 策略5现在已经具备自描述能力：接收端可以从 IP-ID 判断该包属于策略5、属于哪个 path、对应哪个片段顺序；真实隐蔽比特仍由路径排列决定。
- 真实 live 化仍要求 P4/控制面按 `PacketSpec.path_id` 实际调度路径；如果 P4 只是普通轮询而不按策略5生成的 path 序列走，接收端虽然能解析 IP-ID，但路径排列本身就不会对应隐蔽数据。
