# 策略5操作说明

## 1. 策略目标

策略5是“IP-ID 自描述 + 多路径路径序列隐蔽信道”。它利用三条链路的路径排列承载隐蔽信息，同时用 IPv4 Identification 字段中的少量 bit 标明策略号、路径号和片段顺序，方便接收端识别和重组。

## 2. 编码方式

真实隐蔽数据由路径排列表示，一个符号窗口包含 3 个包，窗口内路径排列表示 2 bit：

```text
00 -> path0, path1, path2
01 -> path0, path2, path1
10 -> path1, path0, path2
11 -> path1, path2, path0
```

IP-ID 字段只做自描述，不直接携带真实隐蔽 bit：

```text
bit15      valid = 1
bits14-12 strategy_id = 5
bits11-10 path_id
bits9-0   encrypted fragment_id_mod1024
```

接收端可以从 IP-ID 得到“这是策略5包、属于哪个路径、是第几个承载片段”，再按每三个 `path_id` 组成的排列恢复 2 bit 隐蔽数据。

## 3. 离线验证流程

在虚拟机项目根目录运行：

```bash
cd /home/p4/yws-covert
python3 celue5/run_strategy5_matrix.py
python3 experiments/verify_transmission.py   --strategy 5   --input experiments/data/input_message.txt   --output experiments/results/decoded_s5.bin   --summary experiments/results/verify_s5.json   --trace experiments/results/trace_s5.csv   --pcap experiments/results/packets_s5.pcap
```

运行后重点看：

```bash
cat celue5/results/summary.csv

```

## 4. 当前验证结论

当前已完成策略库级验证：正常顺序、全局乱序都能完整恢复；数据区少量丢包只造成局部 unknown，不会导致后续码流整体错位；帧头区丢包会导致 magic 校验失败，这是当前方案边界。

## 5. live 化说明

策略5当前已经解决“接收端如何知道包属于哪个策略、哪个路径、哪个顺序”的问题：这些信息来自 IP-ID 自描述字段。

但真实 live 化还需要 P4/控制面真正按照 `PacketSpec.path_id` 调度每个承载包。如果 P4 仍然只是普通轮询，那么实际路径序列可能和发送端策略生成的序列不一致，接收端就无法通过路径排列恢复隐蔽数据。后续可做两种方案：

1. 控制面逐包或小窗口下发路径，使承载包按策略5的 `path_id` 序列走。
2. 在 P4 中增加轻量路径提示机制，让策略包按 IP-ID 中的 path_id 选择出口；该方案要注意不要影响普通业务包。
